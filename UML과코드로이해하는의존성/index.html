UML과 코드로 이해하는 의존성

의존성

의존성이란 무엇일까?

우리는 객체지향 프로그래밍을 하면서 의존성이란 의미를 정말 가슴 깊이 받아들이고 있을까?

적어도 난 그렇지 않았던 것 같다.. 그래서 맨날 보면서도 까먹는 의존관계 UML 화살표 방향과 코드로써 의존관계를 명확하게 표현해보면 그래도 좀 덜 잊혀지지 않을까?해서 준비해보았다.

의존성, 추상적으로 다가오는 이 단어를 구체적인 한 문장의 설명으로 이렇게 표현할 수 있다.

A가 B에 의존할 경우, B가 변경될 때 의존하고 있는 A 또한 변경될 가능성이 있음을 의미한다.

변경될 가능성이란 여러가지의 형태가 될 수 있는데 구체적인 예를 들자면 다음과 같다.

  • 클래스의 이름이 변경된다.
  • 메소드의 이름이 변경된다.

  • 클래스의 메소드 구현부가 변경된다. (파라미터, 리턴타입 내부 로직 등)

    • 당연하게도 잘 생각해보면 내부로직 변경의 경우 반드시 A가 영향을 받는 것은 아닐 수도 있다. (설계가 잘 되어있다면)
  • 이 외 무엇이건 변경에 포함되는 것들

이러한 의존성에도 여러 종류가 있고 일반적으로 객체지향에서는 네가지 종류로 분류하고 있다.



Association, 연관관계

// puml @startuml class "A" as a1 class "B" as b1 a1 --r b1 : Association @enduml

UML에서는 아래와 같이 실선으로 나타냄

어떤 객체 A에서 어떤 객체 B로 갈 수 있는 객체간의 그래프 경로가 있음을 의미

여기서의 그래프 경로는 런타임에서의 영구적인 경로를 의미한다. (컴파일이 되고 런라팀에서는 항상 A는 B로 갈 수 있는 경로가 있다.)

마치 코드로 나타내면 다음과 같다.

class A { private B b; }

Association이 객체간의 그래프가 형성되어있는지 유무를 따지는지의 일반적인 관계를 나타내는 것이라고 한다면 더 구체적으로 다음과 같이 관계를 전체와 부분의 집합으로 명확하게 명시하고자 할 때 Aggregation, Composition 으로 나누어서 볼 수도 있다.

(개인적으로 가장 헷갈렸던 부분이다. 블로그나 글을 찾아봐도 차이가 되게 모호하다고 생각했는데, Aggregation, Composition은 Association의 관계표현 종류 중 하나인 것이다.)



Association - Aggregation, 집약관계

// puml @startuml class "A" as a2 class "B" as b2 a2 --ro b2 : Aggregation @enduml

UML로 나타내면 아래와 같이 실선과 칠해지지 않은 다이아몬드(?)로 표현된다.

어떤 객체 A에 어떤 객체 B로 갈 수 있는 객체간의 그래프 경로가 있음을 의미

  • A객체가 소멸되더라도 B객체가 메모리에서 사라지지는 않음
  • 의존관계의 생명주기는 독립적임

마치 코드로 나타내면 다음과 같다.

class A { private B b; public A(B b) { this.b = b; } }

Association - Composition, 합성관계

//puml @startuml class "A" as a3 class "B" as b3 a3 --r* b3 : Composition @enduml

UML로 나타내면 아래와 같이 실선과 칠해진 다이아몬드로 표현된다.

어떤 객체 A에 어떤 객체 B로 갈 수 있는 객체간의 그래프 경로가 있음을 의미

  • 부분 객체가 전체 객체에 속해지는 관계
  • 의존관계의 생명주기 또한 의존적 ( B는 A에 의해서 생명주기가 결정됌)

마치 코드로 나타내면 다음과 같다.

class A { private B b; public A() { this.b = new B(); } }


Dependency, 의존관계

//puml @startuml class "A" as a4 class "B" as b4 a4 ..r> b4 : Dependency @enduml

UML로 나타내면 다음과 같이 점선과 열려있는 화살표로 표현된다.

의존관계의 의존성이라는 것은 다음과 같은 특징중 하나에 속한다.

예) A라는 객체에 B라는 객체의 의존관계가 있을 때의 특징

  • A객체의 어떠한 함수(메소드)에서 B타입의 객체가 파라미터에 명시됌
  • A객체의 어떠한 함수(메소드)에서 B타입의 객체가 리턴타입에 명시됌
  • A객체의 어떠한 함수(메소드)에서 B타입의 인스턴스가 생성됌

마치 코드로 나타내면 다음과 같다.

class A { public void method(B b) { } public B method2() { return new B(); } public void method3() { B b = new B(); } }

Association(연관관계)과 차이가 있다면 Association은 영구적인 협력이 되는 것이고, Dependency(의존관계)는 일시적으로 협력을 필요로 하는 어떠한 시점에 관계를 맺고 끊는 관계이다.



Inheritance, 상속관계

//puml @startuml class "A" as a5 class "B" as b5 a5 --r|> b5 : Inheritance @enduml

UML로 나타내면 아래와 같이 실선과 칠해지지 않은 닫힌 화살표로 표현된다.

상속관계에서도 의존성의 의미가 있는데, 그 이유인즉슨 Super B가 변경될 때 Sub A도 변경될 가능성이 있다라는 것을 의미한다

마치 코드로 나타내면 다음과 같다.

class A extends B { }


Realization, 실체화관계

//puml @startuml class "A" as a6 class "B" as b6 a6 ..r|> b6 : Realization @enduml

UML로 나타내면 아래와 같이 점선과 칠해지지 않은 닫힌 화살표로 표현된다.

말그대로 추상화가 실체화되는 관계이다, 인터페이스와 구현체간의 관계라고 생각하면 이해가 쉽다.

마치 코드로 나타내면 다음과 같다.

class A implements B {}


앞으로도 UML과 코드로 객체간의 의존관계를 명확히 이해해보고 더 좋은 설계에 힘써봐야겠다.


puml 이라고 주석쳐있는 부분의 코드는 plantuml에서 그린 uml 코드이다. 관심이 있다면 plantuml을 찾아보기를 권장한다. IntelliJ에서도 plugin을 지원하는 등 많은 기능을 제공한다. https://plantuml.com/ko/
